Built for real network conditions.

Packet loss, flaky Wi-Fi, and blocked UDP are normal. Onyx keeps the remote side running and gives you a clean way back in.

Handles packet loss Reconnect and resume are part of the normal path instead of a last-resort recovery flow.
Resumes after short drops Interactive shells and foreground exec flows recover across brief transport loss without starting over.
Falls back to SSH automatically Prefer QUIC when UDP works. Use SSH when it does not.
Keeps jobs server-side onyx exec keeps remote work running even when the client disconnects or sleeps.

Stable interactive sessions.

Use Onyx like a normal shell target. It auto-reconnects, keeps the session under tmux, retains remote state server-side, and falls back to SSH when needed.

Interactive shell 
onyx user@host

Connect with QUIC first, fall back to SSH when needed, and use onyx user@host:7373 when the remote listens on a custom port.

Reconnects automatically Short network drops trigger reconnect instead of forcing a fresh login.
tmux-backed Keep shell state and scrollback on the server instead of treating the session as disposable.
Retains the remote session Reconnect to the same shell state instead of opening a fresh session after every brief drop.
Falls back to SSH QUIC is preferred, but the shell path stays usable when UDP is blocked or unavailable.

Run in a remote working directory and stay attached to live output.

Start a job, get a job id immediately, and let it keep running server-side.

Logs onyx logs gpu-box job_xxx

Read buffered output later without taking over the running job.

Kill onyx kill gpu-box job_xxx

Stop a running job explicitly when the work should end.

Resume live output with onyx attach gpu-box job_xxx. For automation, onyx jobs prod --json and onyx exec prod --env RUST_LOG=info --timeout 60s -- cargo test keep state explicit.
