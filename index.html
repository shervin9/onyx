Install the local client

Connection flow

Bootstrap over SSH, prefer QUIC, and keep forwarding or proxy workflows on the same path.

SSH bootstrap
QUIC transport
Forward / proxy
Bootstrap 
onyx user@server

First connect provisions onyx-server on the remote over SSH.

Transport 
onyx user@server

QUIC when available, with SSH fallback if UDP is blocked.

[mode] QUIC
Forward / proxy 
onyx --forward 8888:8888 user@server

Forward local ports over the same QUIC connection, or use onyx proxy %h %p as an SSH ProxyCommand.

More than a shell.

Run commands that survive the network. onyx exec keeps remote work detachable, resumable, reattachable, and scriptable for deploys, training jobs, and agent workflows.

  • Resumable. The server keeps the process alive while your client disappears.
  • Reattachable. onyx attach catches up and streams output again without restarting the job.
  • Detachable. --detach returns a job id, exits cleanly, and lets the work keep moving.
  • Scriptable. --json emits clean NDJSON events for CI, automation, and agents.
Primary flow onyx exec gpu-box --detach -- python train.py

Built for real remote work

QUIC + SSH fallback Prefer QUIC. Fall back cleanly when the network says otherwise.
Session persistence tmux-backed sessions with smoother reconnect behavior and scrollback.
Port forwarding Forward local ports to remote services through the same Onyx connection.
SSH ProxyCommand support Use Onyx as a transport layer for SSH-based tools and existing configs.
Resumable remote exec onyx exec runs commands as jobs that survive client disconnects. Reattach, tail, or capture JSON events later.
AI-friendly workflows Structured --json output and long-lived job ids fit cleanly into agent loops and CI pipelines.

How it works

02 Connect over QUIC

03 Fall back automatically

Switch to SSH when UDP is blocked or unavailable.
